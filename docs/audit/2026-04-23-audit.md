# Security audit — 2026-04-23

- **Scope:** vyakarana pre-1.0 hardening pass. All source files under
  `src/`, the grammar files under `grammars/`, and the smoke / test
  scripts. Reviewed against the 8-point security checklist in
  CLAUDE.md §Hardening step and the AGNOS first-party standards.
- **Auditor:** hardening-pass agent
- **Trust model:** vyakarana is a library over arbitrary input
  bytes (source code). Inputs come from: (a) CLI args passed to
  `vyk`, (b) file contents read into the source buffer, (c)
  grammar file contents read into grammar records. No network I/O,
  no shell-out, no arbitrary writes.

---

## Summary

- **Findings:** 5 LOW, 1 MEDIUM-fixed, 0 HIGH, 0 CRITICAL.
- **Blocking 1.0.0 release:** none. Pending items are defense-in-depth.
- **Fixed during the audit:** 1 (per-token alloc in scanner —
  correctness adjacent to ADR 0002 NFR).
- **Known-CVE-pattern review:** immune by design to the five
  most-relevant classes for parsers / terminal tools (ReDoS,
  unbounded recursion, deserialization-RCE, modeline-escape,
  buffer-overflow-in-parser). Partial exposure on ANSI-escape
  injection via echoed user-argv (FINDING-006, LOW).

## Surface inventory

- 16 `alloc()` call sites (mostly one-time during parse / load)
- 2 `file_read_all()` call sites (`vyk <file>` and `grammar_load`)
- 57 `load8()` call sites
- 0 `syscall(59, ...)` or equivalent exec calls (verified)
- 0 `sys_system()` calls (verified — not in Cyrius stdlib used)

## Findings

### FINDING-001 — per-token alloc in scanner hot path (MEDIUM, fixed)

- **File:** `src/grammars/default_scanner.cyr` line 319 (pre-fix)
- **Severity:** MEDIUM (correctness + ADR violation; not exploitable)
- **Status:** **Fixed during this audit.**

The ident / keyword branch of `tokenize_with_grammar` allocated
8 bytes per iteration via `alloc(8)` to hold an out-parameter slot
for `_ds_lookup_keyword`. Every ident in every tokenized file
therefore triggered a heap allocation — a direct violation of
ADR 0002's "no allocations per token" NFR.

**Fix applied:** changed `_ds_lookup_keyword` to return the kind
directly (i64) with `-1` as the "not found" sentinel; caller
branches on `>= 0`. Gates still green post-fix.

No security exposure — the allocation was benign; the issue is
correctness against a documented NFR.

### FINDING-002 — alloc() return values unchecked (LOW)

- **Files:** all 16 `alloc()` sites in `src/`
- **Severity:** LOW (defense-in-depth)
- **Status:** accepted as-is; documented.

`alloc(N)` can return 0 on OOM. None of the call sites check the
return value before using it as a pointer (`store8`, `store64`,
etc.). A NULL-pointer dereference would segfault on use.

**Exposure:** not an attacker-triggerable condition in the current
threat model — OOM in a tokenizer is a library-fatal event
regardless. A segfault vs. a graceful error path doesn't change
the end state for the consumer (owl, vidya, etc.).

**Why not fixed:** CLAUDE.md §Do-not and the design-spec treat the
tokenizer as a best-effort byte-to-tokens pipeline. Error-path
plumbing for OOM through every `alloc` would bloat the surface
without improving the actual failure mode. If vyakarana later
lands a streaming API (M5) that wants graceful degradation, this
becomes a real concern and gets re-opened.

### FINDING-003 — silent truncation at VYK_SRC_CAP (LOW)

- **File:** `src/main.cyr` line 187
- **Severity:** LOW (correctness; no memory safety issue)
- **Status:** accepted as-is; documented.

`vyk <file>` reads via `file_read_all(path, buf, VYK_SRC_CAP - 1)`
where `VYK_SRC_CAP = 1048576` (1 MB). Files larger than 1 MB are
silently truncated — `file_read_all` caps at maxlen, no warning
is emitted, tokenization proceeds on the prefix.

**Exposure:** no memory safety issue — `file_read_all` honors its
maxlen argument (verified in `lib/io.cyr`). The buffer overrun
risk is zero.

**User-visible consequence:** a 2 MB source file tokenizes its
first 1 MB and emits `}` at whatever the truncation point is. The
coverage invariant still holds over the bytes read. No
`error`-kind tokens emitted unless the truncation lands mid-token
in a way that produces an unrecognized byte sequence.

**Why not fixed:** M5 streaming (iterator API) removes the
fixed-cap entirely. A temporary warning pre-1.0 would be churn.

---

### FINDING-004 — silent truncation at GRAMMAR_FILE_CAP (LOW)

- **File:** `src/grammar.cyr` line 625
- **Severity:** LOW (correctness)
- **Status:** accepted as-is; documented.

Same pattern as FINDING-003: `grammar_load(path)` reads via
`file_read_all(path, buf, GRAMMAR_FILE_CAP - 1)` with cap of 32 KB.
A grammar file larger than 32 KB truncates silently.

**Current reality:** every shipping grammar file is under 3 KB.
Headroom is 10×.

**Why not fixed:** same reasoning as FINDING-003. A size-check +
warning would be two lines; defer until a grammar actually
approaches the cap.

---

### FINDING-005 — Token.start / Token.len are u32 (LOW)

- **File:** `src/token.cyr` line 55 (TOKEN_SIZE), scanner writes
  via `store32`
- **Severity:** LOW (theoretical)
- **Status:** accepted; documented in design-spec §4.

`Token.start` and `Token.len` are u32 per the locked design-spec
layout. A source file larger than 4 GB, or a single token longer
than 4 GB, would overflow on `store32`. Practical file sizes
don't approach this.

**Exposure:** none in the current threat model. Tokens that would
overflow the u32 would fail to round-trip correctly, not create
memory safety issues (the `tokenbuf` slot is still within its
allocation).

**Why not fixed:** u32 spans are a deliberate trade in ADR 0002
(12-byte layout, cache-friendly). Widening to u64 doubles per-token
storage. Keep.

## Clean checks

The following explicit checks produced no findings:

- **Every `load8(src + i + N)` for N > 0** is preceded by an
  `i + N < src_len` bounds check. Verified line-by-line in
  `src/grammars/default_scanner.cyr`, `src/grammar.cyr`,
  `src/grammars/shell.cyr`, and `src/main.cyr`. No OOB reads.
- **No command injection vector.** No `exec`, `sys_system`, or
  equivalent in the codebase.
- **No path traversal vector.** File paths come from argv
  (user-supplied CLI — user owns that) and the hardcoded
  `"grammars/<lang>.cyml"` list in `bootstrap_grammars()`. No
  path segments derived from tokenized content.
- **No integer-overflow on size calculations used for alloc.**
  `_tokenbuf_grow` doubles `byte_cap`; at `2^63`, this would
  overflow. Practically unreachable (would require 2^60 tokens).
- **No TOCTOU on file reads.** `file_read_all` is a single open →
  read → close sequence; we don't re-stat after reading.
- **No untrusted data used as format-string.** All `eprint` /
  `println` / `print` calls pass data as the body argument, never
  as a format specifier (Cyrius has no printf-style formatters
  that would confuse data/format anyway).

## Known-CVE review (2024–2026 patterns relevant to this codebase)

Per CLAUDE.md §Hardening step point 6, the audit checks
recent-CVE patterns for this domain (tokenizers, parsers,
terminal tools). For each pattern, the audit records whether
vyakarana has the same shape and is exposed.

### Regex denial-of-service (ReDoS) — e.g. CVE-2024-52798, CVE-2024-21490, CVE-2024-21503

- **Pattern:** catastrophic-backtracking regex in a parser / lint
  tool, triggered by crafted input.
- **Applicable?** **No.** [ADR 0005](../adrs/0005-m2-rule-type-scope.md)
  explicitly excludes regex from the rule set. The scanner runs
  in O(input bytes) with bounded per-byte work. No backtracking.
- **Status:** immune by design.

### Unbounded recursion in parser — e.g. CVE-2026-33532 (YAML flow-sequence stack overflow), CVE-2026-32141 (flatted), CVE-2025-66031 (node-forge ASN.1), CVE-2025-48924 (Apache Commons Lang)

- **Pattern:** a parser calls itself recursively for each nested
  construct; crafted input with deep nesting exhausts the stack
  with a small payload.
- **Applicable?** **No.** The CYML grammar-loader in
  `src/grammar.cyr` is fully iterative — `_gp_parse` is a single
  `while` loop over the buffer, `_gp_parse_string_array` iterates
  with flat state, no rule invokes itself. The grammar format
  also does not support nested arrays (`[[a,b],[c,d]]` isn't
  syntax), so even a malicious grammar file can't force depth.
- **Status:** immune by design.

### ANSI escape injection — e.g. CVE-2024-52005 (Git sideband XSS), CVE-2025-55193 (Active Record logging)

- **Pattern:** an application echoes untrusted bytes to a
  terminal, letting an attacker-supplied escape sequence
  reposition the cursor, hide text, or forge clickable hyperlinks.
- **Applicable?** **Partially.** vyakarana's token output is
  JSON (NDJSON) — the scanner only emits token kind names (from a
  fixed 10-entry ASCII list) plus start/len integers. Source bytes
  never appear directly in stdout. NDJSON-level ANSI injection
  is not possible.

  However, error messages in `src/main.cyr` (`io_error`,
  `no_grammar_error`, `usage_error`) echo the user-supplied path
  or flag back to stderr without sanitization. A path like
  `$'\e[2J'.sh` would emit the escape to the terminal.
- **Exposure:** the user controls their own argv — an attacker
  passing malicious paths to their own `vyk` invocation attacks
  only themselves. Not a cross-trust-boundary issue. Still,
  stripping control bytes from echoed error args is a
  defense-in-depth improvement for post-1.0. Tracked as
  **FINDING-006** below.
- **Status:** low-priority hardening opportunity, not blocking.

### Buffer overflow in parser / lexer — e.g. CVE-2024-45492 (libexpat), CVE-2024-2961 (glibc iconv), CVE-2024-8748 (libclinkc packet parser)

- **Pattern:** off-by-one, unchecked length, or format-specific
  edge case lets input bytes overwrite adjacent memory.
- **Applicable?** **No identified sites.** The "Clean checks"
  section above documents the bounds-check review — every
  `load8(src + i + N)` for `N > 0` is preceded by an explicit
  `i + N < src_len` guard. `file_read_all` honors its maxlen cap
  (verified in `lib/io.cyr`). `alloc(N)` sizes are either
  constants or derived from already-bounded input lengths.
- **Status:** clean on review; would benefit from a fuzz harness
  post-1.0 (M7 scope per ROADMAP).

### Modeline / in-file sandbox escape — e.g. VIM CVE-2026-34714

- **Pattern:** a file format carries executable configuration
  (modelines, macros) that the tool interprets in the user's
  context.
- **Applicable?** **No.** vyakarana reads source bytes and emits
  tokens. It has no configuration-in-file mechanism, no macros,
  no modelines. Grammar files are data, not code — the CYML
  parser only populates struct fields and never executes anything
  derived from file content.
- **Status:** immune by design.

### Deserialization-to-code-execution — e.g. CVE-2022-1471 (SnakeYAML Java object construction)

- **Pattern:** a data format's "tag" or "type" mechanism lets
  the parser instantiate arbitrary classes, enabling RCE.
- **Applicable?** **No.** Cyrius has no runtime class-instantiation
  machinery; grammar files map to a fixed set of i64 fields and
  cstr pointers. There is no type-dispatch path.
- **Status:** immune by design.

### FINDING-006 — unsanitized path/flag echoed to stderr (LOW)

- **Files:** `src/main.cyr` `io_error`, `no_grammar_error`,
  `usage_error`
- **Severity:** LOW (ANSI escape injection via user's own argv)
- **Status:** accepted; documented.

Per the CVE review above (ANSI escape injection class), vyakarana
error messages echo user-supplied path / flag bytes to stderr
without stripping control characters. A user passing
`file$'\x1b[2J'.sh` to `vyk` would see the escape sequence written
to their terminal.

**Exposure:** the user owns argv — this is a self-attack
scenario. Relevant if a future wrapper ever calls `vyk` with
attacker-influenced paths and forwards stderr to an unprotected
viewer. Low practical risk.

**Fix candidate (post-1.0):** add a `_sanitize_for_stderr` helper
that replaces bytes < 0x20 with `?` before echoing. Applied to
the three error paths. ~10 lines.

## Additional internal-review notes (non-security)

These surfaced during the same review pass; recorded here as
post-1.0 work items rather than audit findings.

1. **Linear scan in `_ds_try_exact_list`** — operators /
   punctuation lookup is O(N) per position. Trie-based longest-match
   (per 2025 tokenizer research on LinMaxMatch) would improve to
   O(match-length). Revisit at M5 benchmark time.
2. **Linear scan in `_ds_lookup_keyword`** — O(K × W) per ident
   where K = word-rule count and W = avg words. For Python with
   ~35 keywords this is bounded. Hash-during-scan or a pre-built
   keyword-table map could make it O(1). Revisit at M5.
3. **Cosmetic token fragmentation** — Rust/C char-literals split
   into three operators, Python f-strings split into ident + string,
   UTF-8 bytes outside strings hit `TK_ERROR` (Markdown stand-in
   sidesteps with ASCII). All preserve coverage + zero-error bars.
   Promote to ADRs post-1.0 if theming work (M4) needs the
   precision.
4. **Regression oracle retained.** `src/grammars/shell.cyr`
   (hand-coded M1) stays on disk until post-1.0. Removing it is a
   one-commit cleanup once we're confident in the data-driven path.
